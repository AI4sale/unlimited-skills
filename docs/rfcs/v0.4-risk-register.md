# v0.4 Risk Register

Status: planning draft. These risks block or constrain v0.4 implementation until owners accept actions and fallbacks.

| ID | Severity | Area | Risk | Owner | Action | Fallback |
| --- | --- | --- | --- | --- | --- | --- |
| R-01 | P0 | Privacy | A recommendation or dashboard flow uploads prompts, task text, skill bodies, private repo data, local paths, tokens, or secrets. | Security/privacy owner | Add redaction tests and payload allowlists before any hosted call. | Disable hosted recommendation/dashboard calls and keep local-only diagnostics. |
| R-02 | P0 | Trust | v0.4 weakens signed hosted manifest requirements or accepts unsigned metadata for decisions. | Security owner | Require signed envelopes and trusted key scopes for recommendations, queues, gates, dashboards, policies, and self-hosted mode. | Treat unsigned data as display-only local fixture data. |
| R-03 | P0 | Autonomy | Future improvement proposals are interpreted as permission for automatic skill rewriting or auto-publish. | Maintainer lead | Keep proposal generation as draft-only and require human review in every state transition. | Remove automatic proposal track from v0.4. |
| R-04 | P0 | Product boundary | Local MIT core becomes registration-gated through recommendation or dashboard work. | Public core owner | Add tests and docs stating local search, list, view, use, feedback, index, and public self-update stay registration-free. | Ship v0.4 modules only as optional registered extensions. |
| R-05 | P0 | Billing | Plan/entitlement work accidentally enables live billing, payment links, checkout, card collection, invoices, refunds, or real charges. | Product/billing owner | Keep billing status sandbox-only and add verifier checks for no live billing authorization. | Omit billing from v0.4 beyond read-only sandbox diagnostics. |
| R-06 | P1 | Release train | Open stacked PRs produce duplicate or conflicting v0.4 work. | Release owner | Enforce PR hygiene gate before each v0.4 batch. | Pause new batches until open release PRs are merged or explicitly blocked. |
| R-07 | P1 | Private registry | Public repo docs assume private registry endpoints not yet accepted or deployed. | Registry owner | Mark private dependencies as contracts until matching private PRs exist. | Keep public client fixture-only. |
| R-08 | P1 | Eval quality | Eval thresholds block useful skills or allow risky skills without transparent override. | Eval owner | Define score bands, coverage requirements, override evidence, and rollback rules. | Keep evals diagnostic-only until thresholds mature. |
| R-09 | P1 | Policy consistency | Enterprise policy, team governance, private packs, catalog browser, and recommendations disagree on allowed actions. | Governance owner | Define a shared policy decision vocabulary and refusal codes. | Refuse ambiguous actions and show diagnostics only. |
| R-10 | P1 | Support safety | Support bundles expose private pack names, private skill names, archive URLs, proofs, keys, or local paths. | Support owner | Keep summaries count-based and add schema/test coverage for forbidden fields. | Exclude v0.4 SkillOps details from support bundles. |
| R-11 | P2 | Self-hosting | Self-hosted registry mode silently falls back to the hosted registry or trusts wrong keys. | Enterprise owner | Require explicit origin/key pinning, no implicit fallback, and diagnostic proof. | Defer self-hosted mode to a later release. |
| R-12 | P2 | User experience | Recommendation output appears to install or update automatically. | UX/docs owner | Use preview-only language and show exact next manual action. | Hide recommendations behind a dry-run flag. |

## Risk Treatment Rules

- P0 risks block v0.4 implementation start.
- P1 risks block release readiness but may allow fixture-only implementation if the release owner accepts the fallback.
- P2 risks can ship as documented limitations only if they do not weaken security, privacy, registration-free local core, or signed manifest boundaries.
