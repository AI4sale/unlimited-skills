# v0.4 Implementation Epics

Status: first implementation plan after readiness blockers are accepted. These epics are ordered to keep the public client fixture-first, signed, redacted, and preview-only.

## Epic 1: SkillOps Contracts And Fixture Gates

Goal: define the signed metadata contracts that every later module consumes.

Scope:

- recommendation manifest schema;
- maintainer queue status schema;
- eval release gate summary schema;
- governance dashboard summary schema;
- self-hosted registry status schema;
- fake-service and fixture-mode responses;
- verifier coverage for no live hosted calls.

Acceptance criteria:

- unsigned metadata is rejected;
- sensitive fields are rejected;
- all payloads are preview-only or read-only;
- support bundle outputs are redacted/count-based;
- local MIT commands remain registration-free.

## Epic 2: Policy-Aware Recommendation Preview

Goal: produce a signed, preview-only recommendation from catalog, eval, improvement, policy, private-pack, org/team, plan, entitlement, and sandbox billing state.

Scope:

- local-only decision table;
- registered signed metadata client;
- shared refusal codes;
- dry-run output;
- explicit hosted query behavior;
- no install/update/remove/write actions.

Acceptance criteria:

- recommendation explains allowed, denied, stale, deprecated, retired, and fixed-pending-eval states;
- public tests prove no prompt, skill body, task text, search query, private path, repo path, token, secret, or private key upload;
- denied actions include owner/action/fallback guidance;
- recommendations never apply changes automatically.

## Epic 3: Eval Gates And Maintainer Queue

Goal: connect measurable quality to human maintainer improvements and release decisions.

Scope:

- eval score bands;
- minimum evidence coverage;
- fixed-pending-eval closure rules;
- queue state transitions;
- release-owner override record;
- changelog/release checklist updates.

Acceptance criteria:

- release gate can block, warn, or pass using signed fixture metadata;
- override requires owner, reason, risk, action, fallback, and expiry;
- queue changes that affect catalog status require maintainer approval;
- no automatic rewrite or auto-publish path exists.

## Epic 4: Governance Dashboard And Self-Hosted Readiness

Goal: expose redacted governance status and prepare optional self-hosted registry mode without moving backend code into the public repo.

Scope:

- dashboard summary model;
- role/permission matrix;
- private-pack redacted counts;
- org/team/policy/plan/billing status aggregation;
- support diagnostic integration;
- self-hosted origin/key pinning diagnostics.

Acceptance criteria:

- dashboard summaries are role-scoped and redacted;
- self-hosted mode requires explicit configuration and trusted keys;
- no hosted fallback occurs unless configured;
- private skill bodies and private inventory remain outside the public repo;
- live billing and full catalog distribution remain out of scope.

## Deferred Track: Human-Reviewed Automatic Improvement Proposals

This track is not an implementation epic for initial v0.4. It may start only after Epics 1 through 4 pass fixture and privacy gates.

Hard requirements:

- proposals are drafts;
- user/admin opt-in is explicit;
- maintainers approve before status changes;
- no automatic skill rewriting;
- no automatic publishing;
- no automatic install/update/remove.
