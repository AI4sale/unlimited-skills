## Summary

The repository already has local learning telemetry and redacted support
surfaces:

- `suggest` records privacy-safe local events and router-call metrics.
- `feedback record` writes local feedback rows after redacting raw query and
  notes fields.
- `learning-summary --events --json` emits aggregate effectiveness and feedback
  counts.
- `feedback prepare` emits a paste-safe support report.
- hosted/catalog skill improvement status exists, but it is a registered
  metadata workflow, not the local v0.6.3 Learning Loop.

The stable v0.6.3 local CLI contract requested by Pro does not exist yet:

- `unlimited-skills learning doctor`: absent.
- `unlimited-skills improvement-candidates`: absent.
- `unlimited-skills apply-candidate --dry-run <candidate-id>`: absent.

## Privacy and Local-Only Boundaries

Current privacy boundaries are strong for the existing surfaces:

- `event_safe_payload` replaces raw `query`, `task`, and `filter` fields with
  summary hashes and presence booleans.
- `event_safe_payload` replaces freeform `notes` with presence and length
  buckets.
- `log_event` writes only to `<root>/.learning/events.jsonl`.
- `feedback record` writes only to `<root>/.learning/feedback.jsonl`.
- `record_router_call` writes only to `<root>/.learning/router-metrics.json`.
- `feedback prepare` explicitly reports `local_only=true`,
  `network_calls=false`, and `upload_available=false`.
- Existing tests assert that raw prompt text, raw query text, raw session ids,
  env values, local absolute paths, skill bodies, `inputSchema`, stdout/stderr,
  and tokens are absent from support and learning outputs.

Status after #173/#175: `feedback record` supports accepted/rejected/neutral
plus the first-class local `missed` and shipped public `wrong` verdicts. Earlier
`wrong-skill` language in this map is historical/conceptual wording; the public
contract uses `wrong`.
