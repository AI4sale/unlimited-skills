# v0.6.8 Verified Completion Memory plan

## Outcome

Unlimited Skills 0.6.8 adds a generic, signed completion-receipt transport to
the public MIT core. An owner-configured private provider may turn an
independently checked and destination-accepted result into durable company
memory. The public core never names or depends on a private knowledge system,
never owns trust keys, and never decides that a completion is accepted.

The release keeps the 0.6.7 retrieval path intact. Retrieval context remains
reference data, not instructions or authority for external action.

## Public contract

- `unlimited-skills context completion-receipt --file receipt.json --json`
  reads a bounded JSON file.
- `unlimited-skills context completion-receipt --stdin --json` keeps the signed
  envelope out of shell history.
- No inline receipt argument exists.
- The client rejects duplicate JSON properties, unknown fields, malformed
  digests, and malformed Ed25519 envelopes before launching the provider.
- Cryptographic verification, key policy, issuer rotation, entity walls,
  transaction state, quarantine, and visibility proof remain provider-owned.
- Status output contains state and identifiers only. It never echoes summary,
  evidence, signature, or the full receipt.

## Stop hook boundary

By contract, the Stop hook never submits prose, URLs, hashes, test counts, cwd scans, tool
logs, or the last assistant message. It accepts only a host-supplied
`trusted_completion_receipt` object or a relative
`trusted_completion_receipt_file` inside the owner-configured inbox.
Possession of either field is not authority: provider-side signature and policy
verification remains mandatory even if an untrusted model can populate it.

Inbox paths are resolved after joining, symlink/junction escape is rejected,
and file size is capped. `UNLIMITED_SKILLS_NO_COMPLETION_MEMORY=1` is the kill
switch. The hook performs enqueue only and has a two-second hard timeout; it
does not wait for atom generation, indexing, or visibility.

## Private transaction contract

The generic public protocol is designed for this state machine without
embedding it in public code:

```text
RECEIVED -> VERIFIED -> PREPARED -> COMMITTED -> PROJECTED -> INDEXED -> VISIBLE
```

Terminal alternatives are `REJECTED`, `QUARANTINED`, `DUPLICATE_VISIBLE`, and
`COMMITTED_NOT_VISIBLE`. A committed record whose index or visibility proof failed is not reported as accepted.
Retrying the same receipt resumes repair.

Receipt identity and logical artifact identity are separate. Immutable
artifacts with conflicting digests quarantine. Mutable file/decision updates
require an explicit `supersedes` record. Existing file-first atoms participate
in dedupe before a new atom is written.

## Client-server compatibility

The stdio provider remains a thin policy/client boundary. It does not read
company files. The same versioned receipt, health, project scope, entity,
sensitivity, provenance, and response states can move from a local loopback
server to a remote company-memory service without changing the agent contract.

The current file-first backend remains supported. A protocol-v2 server may add
an atomic verified-completion overlay while leaving the legacy base index and
daemon untouched. No deletion, replacement, or migration of `library/local`
is permitted by this release.

## Release gates

- Assistant prose with PR/SHA/test-count language causes zero receipt calls.
- A structurally valid signed envelope reaches the configured provider through
  file and stdin transports.
- A malformed signature envelope and duplicate JSON property do not reach it.
- A stalled provider cannot block the Stop hook beyond its fixed budget.
- The exact installed wheel passes both the inherited 0.6.7 retrieval gate and
  the new 0.6.8 receipt-transport gate.
- The private server contract proves Ed25519 policy, SQLite/WAL idempotency,
  concurrent dedupe, legacy atom dedupe, quarantine, crash recovery, and
  `committed/indexed/visible` separation.
- PyPI publication remains OIDC Trusted Publishing from an exact `main` SHA;
  the matching GitHub release is created only after public-index verification.

## Explicit non-goals

- The model never creates its own trusted receipt.
- Raw assistant prose is never acceptance evidence.
- No cross-entity, personal, or restricted-memory write lane.
- No automatic conflict merge or Founder approval per normal verified atom.
- No remote company-memory deployment in this release.
- No claim that `committed` means `visible`.
