# v0.5.0-alpha: PyPI README risk list

Scope: what breaks, misleads, or reads wrong if the **current** repository
`README.md` is published as-is as the PyPI long description for the v0.5
public alpha. Line numbers refer to `README.md` on `main` at the time of
writing (pre-#121 repositioning).

## What is actually configured today

`pyproject.toml` (checked, current state):

- `readme = "README.md"` — hatchling will package the **entire repository
  README** as the long description, with content type inferred as
  `text/markdown`. There is no separate PyPI-facing readme and no
  `dynamic = ["readme"]` setup.
- `version = "0.4.9"` — does not match a v0.5.0 publication; must be bumped
  in the same change that publishes.
- `license = "MIT"` (SPDX-style string) **and** the Trove classifier
  `License :: OSI Approved :: MIT License` are both present. Newer packaging
  toolchains treat license classifiers as deprecated once a license
  expression is set; current twine/hatchling versions may warn or, in strict
  future versions, reject the combination.
- No `[project.urls]` table — the PyPI sidebar will have **no Homepage,
  Repository, Issues, or Changelog links** at all. The only pointer back to
  GitHub would be inside the README body text.
- `[tool.hatch.build.targets.wheel] packages = ["unlimited_skills"]` with
  `force-include` for `packs` — the wheel ships the Python package plus
  bundled `ecc` and `superpowers` packs under
  `unlimited_skills/bundled_packs/`. Repo installer scripts, full docs, and
  examples remain repo assets.

## Risk register

| # | Risk | Severity | Where it bites | Fix |
|---|------|----------|----------------|-----|
| R1 | **Relative links break.** The README contains dozens of relative links: `docs/quickstart.md`, `DONATE.md`, `SERVICE-TERMS.md`, `SECURITY.md`, `CHANGELOG.md`, `LICENSE`, `.github/FUNDING.yml`, every `docs/releases/v0.4.x-alpha*.md` block (lines 175–189), `docs/product-editions.md`, and more. PyPI does not rewrite relative links; every one of them 404s or renders as dead text on the project page. | High | Entire PyPI description; worst in the Install/Status sections where the links carry the actual instructions. | Either rewrite all links as absolute `https://github.com/AI4sale/unlimited-skills/blob/main/...` URLs, or (cleaner) build a short PyPI-facing readme (hatchling metadata hook or a `README-pypi.md` referenced from `readme =`) that links only to absolute URLs. |
| R2 | **In-page anchors break.** `[Install](#install)` (line 508) relies on GitHub's heading-anchor scheme. PyPI's renderer generates different/no anchor ids; the link silently does nothing. | Medium | "Install for Claude Code" section. | Replace anchors with absolute URLs to the GitHub README section or inline the needed command. |
| R3 | **HTML block at the top is sanitized.** `<div align="center">` and `<pre>` (lines 1–19) go through PyPI's readme sanitizer: the `align` attribute is stripped, so the banner renders left-aligned or degrades; if rendering of the whole description fails, PyPI falls back to ugly plain text for the entire page. | Medium | First screen of the PyPI page — the project's first impression. | Use plain Markdown for the banner. Run `twine check dist/*` in CI as a publication gate so a render failure can never ship. |
| R4 | **A3-PYPI-FLIP marker comments become publishable content.** The README carries an HTML comment with the marker (line 501). Sanitizers usually strip HTML comments, but (a) a plain-text render fallback exposes them verbatim, and (b) shipping internal flip markers in a published artifact is leaking release machinery either way. | Medium | Claude Code install section. | The A3 flip itself removes the need: when publishing, flip the install command per the marker protocol and delete the marker comments from the published readme. Verify with a grep for `A3-PYPI-FLIP` over the built sdist/wheel metadata before upload. |
| R5 | **Stale install instructions get frozen into the release.** The README tells users `pip install "git+https://github.com/AI4sale/unlimited-skills.git"` (lines 24, 504) and explicitly says "PyPI packaging is not the supported v0.4.2-alpha distribution path" (line 155) and "the package is not published on PyPI yet" (line 508). On the PyPI page of the published package, every one of these statements is self-contradicting and confusing. | High | Quickstart, Install, Install for Claude Code sections. | The A3 publication gate must flip all marked sites to the plain PyPI command (the `A3-PYPI-FLIP` markers exist precisely to make this mechanical) and delete the "not on PyPI" paragraphs before building the distribution. Add a verifier step: the built long description must not contain `git+https://` install commands or "not published on PyPI". |
| R6 | **Functional claims must match the wheel.** This was the original A3 blocker: `quickstart` promised bundled-pack import while the wheel shipped only `unlimited_skills/`. The v0.5 package branch now force-includes `packs/` into `unlimited_skills/bundled_packs/` and `quickstart` imports from package resources when no checkout is present. Repo installer scripts, full docs, and examples remain repo-only and must be described that way. | High | Quickstart and Status sections read as a description of what `pip install` delivers. | Keep `scripts/run-v050-alpha-package-smoke.py` in the release gate. It must prove wheel bundled-pack inclusion, clean-install `quickstart`, `suggest`, and `mcp savings` before any PyPI flip. |
| R7 | **Claims that read differently out of repo context.** "Pro / Team: planned paid hosted collaboration ..." (line 198), "Business and enterprise access starts with a company registration request" plus a production hostname URL (line 440), donation links in the header and Support section (lines 17, 247), and the long registration/hosted-services section all read, on a PyPI page, like a commercial hosted product listing. v0.5 is a free public alpha with nothing for sale; the page must not look like a storefront. | High | Header, Product Editions, Support, Registration sections. | Land the A2 wording replacement first (see `docs/adoption/pro-team-wording-replacement.md`), and keep the PyPI-facing description to the local-first free core. Keep production service hostnames out of the PyPI description. |
| R8 | **Sheer size and audience mismatch.** The README is ~870 lines and is written for repo visitors of v0.4.x integration gates (the Status section alone lists ~50 gate bullets, lines 63–132). As a PyPI long description it buries the three lines a new user needs. | Medium | Whole page; PyPI also truncates/scrolls poorly on long descriptions. | Short PyPI readme: what it is, install, quickstart, measured numbers, links to GitHub for everything else. The repo README stays the deep document. |
| R9 | **Version drift inside the text.** The README hard-codes "v0.4.9-alpha / developer preview" (line 15) and names v0.4.x gates as current. Published under 0.5.0, the description contradicts the package version on the same page. | Medium | Header and Status sections. | Make the version line part of the release checklist; a verifier should assert the long description mentions the version being published and no newer-sounding stale one. |
| R10 | **Badges and images: none today — keep it that way or use absolute URLs.** The current README has no image or badge markup, so there is nothing to break — but the planned #121 repositioning may add badges. Any future relative image path breaks on PyPI (images must be absolute URLs; PyPI does not serve repo files). | Low (today) | Header, if #121 adds badges. | Rule for #121 and later: badges/images only via absolute `https://` URLs. |
| R11 | **License/metadata presentation.** With both the SPDX `license` string and the license classifier present (see above), upload tooling may emit deprecation warnings; and without `[project.urls]` the page has no source/issues links, which for a security-sensitive tool (it patches `CLAUDE.md`/`AGENTS.md`) reads poorly. | Medium | PyPI sidebar/metadata. | Drop the license classifier (keep `license = "MIT"`), add `[project.urls]` with Homepage, Repository, Issues, and Changelog. |
| R12 | **No render gate in CI.** Nothing currently builds the sdist/wheel and runs `twine check` (there is no `scripts/verify-readme-demo.py` in the repo, and `.github/workflows/ci.yml` does not validate the long description). A render failure or any of R1–R5 would only be discovered on the live PyPI page. | High | The publication itself. | Add to the A3 gate: build, `twine check dist/*`, plus greps over the built METADATA for `git+https://`, `A3-PYPI-FLIP`, relative `](docs/` links, and the dead-command rule from `tests/test_install_path_docs.py`. |

## Notes

- `tests/test_install_path_docs.py` already enforces that the literal
  unpublished install command never reappears without an `A3-PYPI-FLIP`
  marker on the same line; the A3 flip is designed to be mechanical
  (grep the marker, flip each site, delete the marker).
- PR #121 repositioned the README and #123 added this risk list. Re-check
  R3/R8/R10 against the final README before building the published
  description, and re-run the long-description greps either way.
