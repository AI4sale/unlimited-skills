# v0.4 Readiness Audit

Status: development-ready planning draft, no-go for implementation start until the blockers below are assigned and accepted by the release owner.

`v0.4` should move Unlimited Skills from `v0.3.x` feature maturity into a governed SkillOps platform only after the public core, private registry, catalog quality signals, maintainer workflow, policy surfaces, and release train are coherent enough to support measured skill delivery. This audit is planning/specification only. It does not implement runtime behavior, enable live billing, publish to PyPI, distribute the full catalog, rewrite skills, or publish skills automatically.

## Go/No-Go Recommendation

Recommendation: **NO-GO for v0.4 implementation start until P0 blockers are closed**.

The `v0.3.9-alpha` stack proves the needed ingredients exist in alpha form: signed hosted metadata, catalog browser, explicit feedback, eval quality signals, maintainer-controlled improvement status, private/team pack client flows, org/team governance diagnostics, plan and sandbox billing diagnostics, support diagnostics, and release verifiers. `v0.4` can start after `v0.3.9` only when those ingredients have a single architecture, ownership model, policy vocabulary, and release gate. The release owner may approve parallel discovery spikes, but not runtime implementation that changes hosted distribution, billing, telemetry, or automatic improvement behavior.

## Explicit Non-Readiness Statements

- v0.4 is not approved for implementation by this document.
- v0.4 is not ready for release or production hosted rollout; no live billing, no PyPI publication, and no full catalog distribution are authorized.
- v0.4 does not authorize automatic rewriting, automatic telemetry, or automatic hosted query forwarding.
- v0.4 has a no auto-publish boundary: generated or accepted improvements require human-reviewed publication.
- v0.4 must not gate the MIT local core behind registration.
- v0.4 must not put private skill bodies, private pack inventory, secrets, or hosted backend code into the public repo.
- v0.4 must not weaken signed hosted manifest requirements in recommendation, queue, gate, dashboard, policy, or self-hosted mode.

## Entry Criteria For v0.4 Implementation

- All P0 blockers have owners, accepted contracts, testable acceptance criteria, and fallbacks.
- The public client contract preserves no-prompt, no-skill-body, no-private-data, no-token, no-private-key, and no-full-path boundaries.
- Signed hosted manifest requirements remain mandatory for hosted metadata that affects recommendations, release gates, queue status, policies, or install decisions.
- Release owner signs the v0.4 PR hygiene gate after `#65` and `#66` are resolved or explicitly blocked.
- The first implementation epic starts with fixture-mode tests and no production hosted calls.
