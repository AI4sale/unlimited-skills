# Unlimited Skills v0.2.1-alpha RC

This release-candidate branch integrates the Local Skill Hub runtime stack that was previously split across PR #9, PR #10, PR #11, and PR #12.

## Highlights

- Local Skill Hub protected `/v1/...` APIs require a valid, non-revoked hub client token.
- `GET /health` remains unauthenticated for local liveness checks.
- Hub tokens can be created, listed, and revoked with `unlimited-skills hub token create|list|revoke`.
- Hub tokens are stored as SHA256 hashes; raw token values are shown only once at creation.
- LAN bind remains opt-in: non-localhost hosts require `--allow-lan` and at least one active hub token.
- Remote client runtime is implemented for `remote configure`, `remote status`, `remote search`, `remote resolve`, and `remote view`.
- Remote fallback policy is explicit: `local_allowed` can use the local library when the hub is unavailable; `hub_required` fails closed.
- Hub allowlist bootstrap can cache a validated local fixture allowlist or registered hosted allowlist metadata.
- The v0.2.x smoke suite now runs real hub token, remote client, and allowlist bootstrap scenarios on the integrated branch.
- Agent installers can render remote-first router instructions for Codex, Claude Code, Hermes, and OpenClaw.

## Product Boundary

- Existing `unlimited-skills serve` remains the free MIT local daemon and does not require registration.
- `unlimited-skills hub serve` remains registration-gated.
- Local search, list, view, where, reindex, vector-reindex, adapt, native sync, installers, migration scripts, and public self-update remain local-first and unregistered.
- Local Skill Hub remains alpha and allowlist-only.
- Full catalog distribution remains disabled.
- The launch policy follows the private registry audit verdict `YES_WITH_ALLOWLIST`.
- Hosted registry services do not receive local hub search queries by default.
- The hub does not execute skills.
- Remote-first installer mode writes local hub config and router instructions only. It does not registration-gate local MIT commands and does not store raw hub tokens in visible router files.

## Security Notes

Current client-side hosted archive security still relies on SHA256 verification and safe extraction. Cryptographic signature verification for hosted manifests and allowlists is planned for a later release and must not be claimed as implemented in v0.2.1-alpha.

For serious LAN deployment, run the hub behind reverse proxy or network controls that provide TLS, authentication, access logging, and IP allowlisting.

## Verification

Required release-candidate checks:

```text
python -m pytest tests -q
python scripts/run-v0.2x-smoke-tests.py
git diff --check
```

The smoke suite must use a temporary HOME, must not call production hosted registry services, and must not expose raw hub tokens, registration tokens, or device private keys in captured output.
