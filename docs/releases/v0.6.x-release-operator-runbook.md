# v0.6.x Release Operator Runbook

Status: operator runbook for future `v0.6.x` patch releases.

This runbook is the repeatable release path for `v0.6.x` patch releases after
the `0.6.0` uploaded-but-not-released incident. It is documentation only: it
does not publish a package, upload to PyPI, create a tag, yank a package,
submit a marketplace listing, change runtime behavior, reopen #119/E19, add
telemetry, add paid CTAs, or claim hosted, team, or enterprise readiness.

## 1. Release Purpose And Allowed Scope

A `v0.6.x` patch release may only promote the public core when the frozen
v0.6 CLI, stdout JSON, local privacy, feedback, ROI receipt, packaging, and
release verification contracts still pass.

Allowed release work:

- select the exact `main` commit;
- run local and CI-safe verification;
- publish through GitHub Actions PyPI Trusted Publishing after approval;
- smoke-test the real PyPI package;
- run the published-mode verifier;
- create the release tag and GitHub prerelease only after published-mode pass
  evidence exists.

Disallowed in this runbook:

- no actual v0.6.2 release without an explicit release task;
- no local PyPI token, `.pypirc`, `TWINE_PASSWORD`, or manual upload;
- no PyPI upload/yank from this documentation PR;
- no retroactive tag for a failed package;
- no PyPI yank action unless the owner explicitly approves that mutation;
- no marketplace submission;
- no #119/E19 distribution work;
- no paid CTA, payment link, checkout path, support SLA, delivery promise, ROI
  guarantee, hosted readiness claim, team readiness claim, or enterprise
  readiness claim;
- no telemetry, upload, analytics SDK, tracking pixel, hosted query
  forwarding, prompt collection, tool input collection, or tool output
  collection.

## 2. Preconditions

Before release work starts:

| Check | Required result |
| --- | --- |
| Release owner approval | explicit approval for the release cleanup/publish turn |
| Branch | local `main` refreshed from `origin/main` |
| Open public PRs | #119 may remain open only as parked E19; release PRs must be merged or blocked |
| Package version | intended patch version is committed before publish |
| CI | release PR CI is green before merge |
| Frozen contract harness | passes before publish |
| Release verifier | passes in prepublish mode before publish |

Use:

```bash
git checkout main
git pull --ff-only origin main
MAIN_SHA="$(git rev-parse HEAD)"
```

## 3. Prepublish Verification

Run the local release gate from the exact selected `main` commit:

```bash
python scripts/verify-v06-frozen-contracts.py --json

python scripts/verify-v060-alpha-publication.py \
  --expected-sha "$(git rev-parse HEAD)" \
  --package-availability prepublish \
  --json

python -m pytest tests/test_public_alpha_docs.py -q
python scripts/verify-mcp-boundaries.py
python scripts/verify-feedback-report-boundaries.py
python scripts/verify-roi-receipt-boundaries.py
python -m pytest tests -q
git diff --check
```

The prepublish verifier must keep tag guidance blocked until a real PyPI
upload, clean package smoke, and published-mode verifier pass have all
completed.

## 4. Frozen-Contract Harness Requirement

`scripts/verify-v06-frozen-contracts.py --json` is mandatory before publish and after publish.
The release verifier invokes it in both prepublish and published modes.

The harness must cover:

- `unlimited-skills --version`;
- `unlimited-skills quickstart --json`;
- `unlimited-skills suggest "Design a REST API for a service" --json`;
- `unlimited-skills mcp install --claude-code --dry-run`;
- `unlimited-skills mcp savings --json`;
- `unlimited-skills feedback prepare --json`;
- `unlimited-skills learning-summary --events --json`;
- `unlimited-skills roi receipt`;
- `unlimited-skills roi receipt --format json`;
- `unlimited-skills roi receipt --since 7d`;
- `scripts/generate-public-alpha-signal-rollup.py --fixture-mode --out <tmp>`.

If the frozen-contract harness reports drift or a blocked surface, the release
is blocked. Do not publish, tag, or create a GitHub prerelease until the drift
is resolved or an explicit owner decision records a different action and
fallback.

## 5. Trusted Publishing Dispatch Inputs

Use the GitHub Actions workflow only:

```text
workflow: publish-pypi.yml
environment: pypi
version: <version>
expected_sha: <MAIN_SHA>
confirm_pypi_publish: publish unlimited-skills <version> to PyPI
```

Do not use local PyPI credentials. Do not use a local `.pypirc`. Do not set
`TWINE_PASSWORD` or `PYPI_TOKEN`.

## 6. Real PyPI Install Smoke

After the Trusted Publishing workflow succeeds, create a fresh environment and
install the exact package from PyPI:

```bash
python -m pip install --upgrade pip
python -m pip install --no-cache-dir unlimited-skills==<version>
unlimited-skills --version
unlimited-skills quickstart --json
unlimited-skills suggest "Design a REST API for a service" --json
unlimited-skills mcp install --claude-code --dry-run
unlimited-skills mcp savings --json
unlimited-skills feedback prepare --json
unlimited-skills learning-summary --events --json
unlimited-skills roi receipt
unlimited-skills roi receipt --format json
unlimited-skills roi receipt --since 7d
```

The installed package must accept `learning-summary --events --json` and the
ROI receipt JSON must validate without forbidden private fields.

## 7. Published-Mode Verifier

Run the published-mode verifier only after the real PyPI smoke passes:

```bash
python scripts/verify-v060-alpha-publication.py \
  --expected-sha "$MAIN_SHA" \
  --package-availability published \
  --json
```

The published-mode verifier must pass and must include frozen-contract pass
evidence before any release tag or GitHub prerelease is created.

## 8. Tag And GitHub Prerelease Sequence

Only after the published-mode verifier passes:

```bash
git tag -a v<version>-alpha "$MAIN_SHA" -m "v<version>-alpha"
git push origin v<version>-alpha
```

Then create the GitHub prerelease against that tag. Release notes must identify
the package version, tag target, PyPI workflow evidence, real PyPI smoke
evidence, published-mode verifier result, privacy boundaries, and any known
incident or replacement-package note.

## 9. Hard Stops

Stop immediately and do not tag or create a GitHub prerelease if any item
fails:

| Stop condition | Owner | Action | Fallback |
| --- | --- | --- | --- |
| Frozen-contract harness fails | release owner / maintainer | Fix drift or document a blocker with command evidence | Keep release untagged and open a patch blocker |
| Prepublish verifier fails | release owner / maintainer | Fix release-gate evidence, SHA, version, or docs drift | Do not dispatch PyPI workflow |
| Trusted Publishing workflow fails | release owner | Inspect workflow logs and fix package/workflow issue | Do not retry blindly; keep release untagged |
| Real PyPI install smoke fails | release owner / maintainer | Record failing command and fix in a new patch version | Do not tag the uploaded artifact |
| Published-mode verifier fails | release owner / maintainer | Record verifier output and prepare a hotfix version | Do not tag or create a prerelease |
| ROI receipt JSON leaks forbidden fields | privacy owner / maintainer | Fix schema/output boundary and rerun verifier | Block release and document privacy blocker |
| Docs claim paid/hosted/team/enterprise readiness | release owner / docs owner | Remove unsupported claims and rerun docs/security checks | Block release until docs are corrected |
| #119/E19 is included in release target | release owner | Remove distribution-lane work from target | Keep #119 parked |

Every blocker must have owner/action/fallback before the release task can close.

## 10. Incident Fallback: Uploaded But Verifier Failed

The 0.6.0 incident is the reference failure class:

- `unlimited-skills==0.6.0` was uploaded to PyPI;
- the package was uploaded but not tagged;
- the published CLI rejected `learning-summary --events --json`;
- the published verifier failed;
- no `v0.6.0-alpha` tag was created;
- no GitHub prerelease was created;
- `unlimited-skills==0.6.1` became the valid replacement.

If a future package is uploaded but fails post-publish smoke or the
published-mode verifier:

1. Mark the package as uploaded but not released.
2. Do not create a retroactive tag.
3. Do not create a GitHub prerelease.
4. Open a hotfix version because PyPI cannot replace an uploaded artifact.
5. Link the incident from release notes, compatibility docs, and the compliance
   audit.
6. Record owner/action/fallback for whether the bad uploaded package should be
   yanked. Do not perform a yank without explicit owner approval.

## 11. Owner/Action/Fallback Table

| Area | Owner | Action | Fallback |
| --- | --- | --- | --- |
| Release approval | release owner | Approve merge/publish/tag window explicitly | Keep release PR open |
| Package upload | release owner | Dispatch Trusted Publishing with exact inputs | Keep package unpublished |
| Frozen contracts | maintainer / QA automation | Run harness before and after publish | Block release and open drift blocker |
| Privacy boundaries | privacy owner / maintainer | Run feedback and ROI boundary verifiers | Block release until unsafe output is fixed |
| Failed uploaded artifact | release owner / PyPI project owner | Decide yank yes/no and record reason | Leave untagged/unreleased and point users to replacement |
| #119/E19 | product owner | Explicitly reopen distribution lane if desired | Keep #119 parked and excluded |

## 12. #119/E19 Exclusion Rule

#119/E19 is parked distribution-lane work. It must not be merged into a v0.6.x
release, included in release evidence, or used as proof of public-alpha
readiness unless the owner explicitly reopens that lane. A valid v0.6.x release
report should show #119 as parked or absent from the release target.

## 13. No Paid/Hosted/Team Claim Boundary

v0.6.x public-alpha releases do not claim:

- paid plan readiness;
- payment link or checkout path readiness;
- hosted service readiness;
- team mode readiness;
- enterprise readiness;
- marketplace acceptance;
- support SLA;
- delivery promise;
- guaranteed ROI or guaranteed savings.

Release notes may describe measured local behavior and verified local public
core contracts only.

## Evidence Checklist

For every v0.6.x patch release report, include:

- selected `MAIN_SHA`;
- PR merge evidence and branch cleanup;
- `python scripts/verify-v06-frozen-contracts.py --json` pass output;
- prepublish verifier output;
- PyPI Trusted Publishing workflow URL;
- real PyPI install smoke output;
- published-mode verifier output;
- tag target and GitHub prerelease URL after verifier pass;
- owner/action/fallback table for any blocker;
- #119 parked proof.
