# v0.3.1-alpha Release Health

Status: in review.

## Install Status

Fresh install is verified by `scripts/run-v0.3.1-alpha-post-release-smoke.py`. The smoke installs the checkout in an isolated temporary HOME, indexes a local skill, searches it, and verifies that no production hosted calls are required.

## Upgrade Status

The stabilization smoke verifies:

- synthetic upgrade from a v0.2.2-style `local/` and `registry/` library;
- synthetic upgrade from a v0.3.0-style `local/` and `registry/` library;
- no destructive mutation of existing `local/` or `registry/` skill files during reindex.

## Core Local Commands

Local `reindex`, `list`, `where`, `search`, and `view` remain registration-free in the smoke path. The MIT local core is not gated by hosted-service registration.

## Registered Feature Status

Hosted catalog, updates, Local Skill Hub registry sync, team sync, and managed Enterprise Skill Lock sync remain registration-gated early-access features.

## Known Limitations

See `docs/known-limitations.md` and `docs/releases/v0.3.1-alpha.known-issues.md`.

## Support And Debug Paths

- `unlimited-skills doctor --json` remains the local diagnostic entry point.
- `service doctor`, `service verify-trust`, `service test-registration --dry-run`, and `service test-proof` remain the registered-service diagnostics.
- `support bundle` produces a redacted support diagnostic bundle for operator review without sending skill bodies, prompts, local paths, tokens, private keys, or environment values by default.
- diagnostics v2 checks are connected to setup and support workflows with explicit network checks and redacted output.

## Private Registry Reconciliation

Private registry reconciliation remains in review until `AI4sale/unlimited-skills-registry` PR #10 is accepted. The publication manifest records the canonical 315-skill audit, with 105 skills in the Hub allowlist, 165 skills requiring local install plans, and 45 excluded skills.

## Production Registry Signing

Production-signed registry artifacts are not verified yet. The final publication verifier fails by default until the protected registry signing ceremony reports production-signed artifacts, or until the release owner explicitly accepts blocked registry signing as a v0.3.1-alpha known issue.

## Open Risk Register

- PyPI remains deferred until package asset inclusion and installer behavior are tested in CI.
- Full catalog distribution remains disabled until allowlist, blocked, local-only, needs-review, and tool-dependency classes are governed.
- Archive bytes are SHA256-verified and safely extracted, but archive-byte cryptographic signatures are not implemented in this alpha.
- Hosted registry features require registered installation, trusted signed manifests, and valid device proof.
- Production-signed registry artifact completion is still a release-owner decision point.
