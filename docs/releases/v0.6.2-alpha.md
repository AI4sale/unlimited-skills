# v0.6.2-alpha

Status: prepared release — critical fix for non-English users. (Source is
marked 0.6.2; the PyPI publish + "accepted release" doc flip are the
release-operator steps below.)

`v0.6.2-alpha` fixes a critical retrieval bug: **non-English users got zero
skills back.** The `suggest` probe is lexical-only with an ASCII-only
tokenizer, so a prompt in any non-Latin language (Russian, Chinese, Arabic, …)
tokenized to nothing, scored zero, and returned no skills — the router failed
silently for every non-English user, while agents that emit English keywords
(Codex/Hermes) kept retrieving correctly. This is a release on its own: without
it the product is broken for non-English users.

## What changed

- **Three-tier non-English rescue** (in `suggest` + the `UserPromptSubmit` hook):
  1. cheap lexical retrieval for English (unchanged fast path);
  2. on a non-English / lexical-empty prompt, route to the local **multilingual
     embedding sidecar** when it is installed;
  3. if no sidecar is installed (or it is too cold to answer within the probe
     timeout), inject an instruction telling the model to restate the task as
     3-8 English keywords and re-query — **never silence**. That instruction
     also warns that native-language vector search needs a warm daemon (a cold
     lookup costs ~14s) and tells the model to **ask the user to approve**
     starting `unlimited-skills serve` (never auto-started).
- **Router-call counter** at `<root>/.learning/router-metrics.json` (total
  invocations, last-call timing/path/outcome, per-day histogram), surfaced in
  `unlimited-skills doctor`. Privacy-preserving: skill names, numeric scores,
  and outcome codes only — never query text or filesystem paths.
- **Multilingual install guidance**: the managed CLAUDE.md router block now
  tells the assistant to default to the **vector + daemon** install
  (`vector-reindex` + `serve`) whenever it has ever conversed with the user in
  a language other than English.

The new `retrieval_path` / `needs_english_query` fields ride only the `--card`
channel, so the plain `suggest --json` contract is unchanged.

## Install (after publish)

```bash
python -m pip install --upgrade unlimited-skills
unlimited-skills quickstart
```

Non-English users should also build the multilingual sidecar and keep the model
warm so tier 2 stays fast:

```bash
unlimited-skills vector-reindex   # multilingual embedding sidecar
unlimited-skills serve            # warm daemon
```

## Release-operator steps (publish gate)

These are intentionally NOT done in the fix PR — they belong to the publish
gate (see `v0.6.x-release-operator-runbook.md`):

1. Bump the publish workflow guard + confirm string to `0.6.2`
   (`.github/workflows/publish-pypi.yml`) and update the package-smoke /
   publication-verifier steps to a `v062` variant.
2. Run Trusted Publishing (manual `workflow_dispatch`) → real PyPI upload.
3. Run the published-mode verifier against the live PyPI package.
4. Only then flip the "accepted release" docs (README, README-pypi, public
   alpha docs) from 0.6.1 to 0.6.2 — publish first, claim accepted second
   (the v0.6.0 uploaded-not-released lesson).
5. Tag `v0.6.2-alpha` and create the GitHub release.

## Boundaries

- No hosted, team, or enterprise readiness claim is made by this release.
- No telemetry, upload, analytics, hosted query forwarding, prompt collection,
  or tool input/output collection is added.
- The multilingual sidecar and daemon run entirely locally.
- #119/E19 remains parked unless explicitly reopened.
